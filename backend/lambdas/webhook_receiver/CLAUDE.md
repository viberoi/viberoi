# CLAUDE.md — webhook_receiver Lambda

Receives inbound webhooks from GitHub / GitLab / Linear via API Gateway HTTP API v2, verifies HMAC, looks up the org by the per-integration path UUID, pushes raw payload to SQS `webhook_events` for the Worker.

Never processes the body. Never decodes JSON. Verify → publish → 200.

## URL shape

```
POST /webhooks/{provider}/{integration_id}
       │            │
       │            └─ UUID, looked up in integration_oauth_tokens
       └─ github | gitlab | linear
```

Customer-facing URLs are generated by the Integration service when an OAuth flow completes. Each `(org, provider)` gets one integration row → one URL → one webhook secret.

## Handler flow

```
1. lambda_auth.verify(event, context, expected_source="webhook:<provider>")
   — sanity-check the API-Gateway-HTTP-API-v2 shape.

2. Parse path: provider + integration_id (404 on malformed).

3. Load raw body — base64-decoded if isBase64Encoded.

4. superuser_session() → look up integration_oauth_tokens row,
   confirm provider matches, decrypt webhook_secret via crypto.envelope
   (AAD: org:{org_id}:integration:{provider}:field:webhook_secret).
   (404 on revoked / not found / wrong provider.)

5. viberoi_shared.webhooks.verify(provider, headers, raw_body, secret).
   (401 on signature mismatch.)

6. Extract delivery_id via viberoi_shared.webhooks.extract_delivery_id().

7. viberoi_shared.sqs.publish("webhook_events", {
     "org_id": ..., "provider": ..., "delivery_id": ...,
     "headers": {filtered}, "body_b64": ..., "received_at": ...
   }).

8. Return 200 immediately. The Worker processes async.
```

## Why path-based integration_id

GitHub/GitLab/Linear webhooks don't carry org-identifying headers — they just post to the URL the customer configured. So the URL itself has to identify the integration. We use the `integration_oauth_tokens.id` UUID in the path because it's already unique per (org, provider) and we can look up the org + secret from it.

A new integration → new UUID → new URL. Customer copies the new URL into the provider's webhook UI when rotating.

## Status code semantics

- `200`: payload accepted, pushed to SQS. Provider stops retrying.
- `401`: HMAC mismatch / lambda_auth failed / integration revoked.
- `404`: malformed path / integration_id doesn't exist / provider mismatch.
- `500`: bug. Provider will retry per its own policy (GitHub backs off exponentially).

We never return a 4xx for transient errors (DB unavailable, SQS unavailable) — those go 500 so the provider retries.

## Rules that apply

- Never log the body. Never log the webhook secret. Logs carry `request_id`, `provider`, `integration_id`, `delivery_id`, and the outcome — that's it.
- Never `import boto3` directly (use `viberoi_shared.aws.*` / `viberoi_shared.sqs.*`).
- Never decrypt the secret outside `viberoi_shared.crypto.envelope`.
- `lambda_auth.verify` is the first line of logic. Always.
